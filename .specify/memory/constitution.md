# user-service Constitution

## Core Principles

### I. N-Layer Architecture (NON-NEGOTIABLE)

Every feature MUST be implemented across the following discrete layers, each
in its own package, with strict, one-directional dependencies (top → bottom):

1. **controller** — HTTP entry points; request/response orchestration only.
2. **service** — business logic; transactional boundaries.
3. **domain** — entities, value objects, and domain rules.
4. **repository** — persistence contracts and implementations.
5. **dto** — request/response payloads exchanged at controller boundary.
6. **exceptions** — domain and application exceptions plus a centralized
   handler (e.g., `@RestControllerAdvice`).

Rules:
- Controllers MUST NOT contain business logic and MUST NOT depend on
  repositories directly.
- Services MUST NOT return domain entities to controllers; they MUST return
  DTOs (or be mapped at the controller boundary by a dedicated mapper).
- Domain MUST NOT depend on Spring framework annotations beyond what is
  strictly required for persistence (e.g., JPA annotations are acceptable
  on entities; web annotations are forbidden in domain).
- Cross-layer leakage (e.g., DTOs imported in repository, JPA entities
  exposed in controller responses) is a blocking violation.

**Rationale**: A clear N-layer split is the contract that makes the rest of
the constitution enforceable. Without it, AOP logging, versioning gates,
and SOLID review have no stable seams to attach to.

### II. Test Discipline: Unit + Functional (NON-NEGOTIABLE)

Every feature MUST ship with both:

- **Unit tests** for service-layer logic and any non-trivial domain rule,
  using JUnit 5 + Mockito. External collaborators (repositories, clients)
  MUST be mocked. Each public service method MUST have at least one happy-
  path test and one failure-path test.
- **Functional tests** that exercise the HTTP boundary end-to-end with the
  Spring context loaded (e.g., `@SpringBootTest` + `MockMvc` or
  `WebTestClient`, or `@WebMvcTest` for controller slices when persistence
  is not under test). Functional tests MUST cover every exposed endpoint,
  including at least one success and one error response.

Rules:
- A pull request that introduces or modifies a controller, service, or
  domain rule without corresponding unit AND functional tests MUST be
  rejected at review.
- Tests MUST live under `src/test/java` mirroring the production package
  structure; unit tests in `*.unit` or co-located, functional tests in a
  `*.functional` (or `*.e2e`) package — choose one convention per module
  and apply it consistently.
- Build MUST fail when tests fail (`tasks.named('test')` is the gate).

**Rationale**: Unit tests catch logic regressions cheaply; functional tests
catch wiring, serialization, and contract regressions that unit tests
cannot. Both layers are required because each covers a class of defects
the other cannot.

### III. API Versioning (NON-NEGOTIABLE)

Every public HTTP endpoint MUST be versioned. The default mechanism is
URI path versioning with the prefix `/api/v{N}/...` (e.g.,
`/api/v1/users`). Rules:

- A new endpoint MUST start at `v1` and MUST NOT be exposed without a
  version segment.
- Breaking changes (removed/renamed fields, changed types, changed status
  codes, changed authentication semantics) MUST introduce a new major
  version (`/api/v2/...`). Non-breaking additive changes MAY remain in the
  current major version.
- A deprecated version MUST be marked with the `Deprecation` HTTP header
  and a `Sunset` header (RFC 8594) and MUST NOT be removed in the same
  release that introduces its successor.
- Version constants (e.g., a `ApiVersions` class or `@RequestMapping`
  base path) SHOULD be centralized so a single change updates all
  references.

**Rationale**: Versioning is the only contract that lets clients evolve
independently of the service. Treating it as optional creates silent
breakages that this project cannot afford.

### IV. Centralized Logging via AOP (NON-NEGOTIABLE)

Business code (services, domain) MUST NOT contain logging statements
(`log.info`, `log.debug`, `log.error`, `System.out`, etc.) for purposes
of tracing method entry/exit, arguments, return values, or execution
time. All such logging MUST be implemented as Spring AOP aspects in a
dedicated `aop` (or `logging`) package.

Rules:
- Required aspects, at minimum: a controller-entry/exit aspect, a
  service-entry/exit + duration aspect, and an exception-logging aspect
  bound to the centralized exception handler.
- Log format MUST be structured (key=value or JSON) and MUST include a
  correlation/trace id propagated per request.
- The single permitted exception is logging *inside* the AOP aspects
  themselves and *inside* the centralized exception handler. Any other
  logger usage in business code is a blocking violation.
- A static check (Checkstyle/ArchUnit/PMD or a CI grep gate) SHOULD be
  added to enforce this; until automated, code review MUST enforce it.

**Rationale**: Logging mixed into business logic is the leading source of
noise, drift, and inconsistent observability. Centralizing via AOP keeps
business code readable and ensures every method gets the same treatment.

### V. SOLID, DRY, YAGNI (NON-NEGOTIABLE)

All production code MUST observe:

- **SOLID**: Single Responsibility (one reason to change per class),
  Open/Closed (extend via new types, not by editing stable ones),
  Liskov (subtypes substitutable), Interface Segregation (small,
  client-specific interfaces), Dependency Inversion (depend on
  abstractions, inject via constructor — no field injection of
  collaborators).
- **DRY**: Knowledge expressed once. Duplicated logic across services or
  controllers MUST be extracted to a shared component. Three identical
  occurrences is the threshold that forces extraction; two is a warning.
- **YAGNI**: Build only what the current story requires. No speculative
  abstractions, no "for future use" parameters, no configuration flags
  without a present consumer.

Tension between DRY and YAGNI is resolved in favor of YAGNI when the
abstraction is speculative; in favor of DRY when the duplication is
proven (≥3 sites) and the abstraction is obvious.

**Rationale**: These principles are the day-to-day vocabulary of code
review. Naming them here makes "this violates SRP" or "this is YAGNI" a
constitutional argument rather than a stylistic opinion.

## Technology & Architectural Constraints

- **Runtime**: Java 25 (toolchain pinned in `build.gradle`).
- **Framework**: Spring Boot 4.0.x. Upgrades within 4.0.x are allowed in
  PATCH PRs; major framework bumps require a MINOR constitution update if
  they affect any principle (e.g., logging or versioning conventions).
- **Build**: Gradle (wrapper checked in). Builds MUST be reproducible from
  a clean checkout via `./gradlew build`.
- **Package root**: `com.cat.user.service`. Layer packages MUST be
  immediate children: `com.cat.user.service.controller`,
  `...service`, `...domain`, `...repository`, `...dto`,
  `...exceptions`, `...aop`.
- **Lombok** is permitted for boilerplate (`@Getter`, `@Builder`,
  `@RequiredArgsConstructor`); `@Slf4j` is permitted ONLY in `aop` and
  `exceptions` packages (see Principle IV).

## Development Workflow & Quality Gates

- Every change is delivered through a pull request. Direct pushes to the
  default branch are forbidden.
- A PR MUST be blocked from merge unless ALL of the following hold:
  1. `./gradlew build` succeeds (compile + unit + functional tests).
  2. New/changed endpoints are versioned per Principle III.
  3. New/changed services have unit AND functional tests per Principle II.
  4. No `log.*` calls were added outside `aop`/`exceptions` packages.
  5. The diff respects the layer boundaries of Principle I.
- Code review MUST explicitly call out which principle a change touches.
  "Looks good" is not an acceptable approval comment for any PR that
  introduces or alters a controller, service, or domain rule.
- Spec/plan/tasks artifacts produced by the `/speckit.*` commands MUST
  reference this constitution by version in their "Constitution Check"
  sections.

## Governance

This constitution supersedes ad-hoc conventions, individual preferences,
and prior informal agreements. In any conflict between this document and
other guidance, this document wins until it is amended.

**Amendment procedure**:

1. Open a PR that modifies `.specify/memory/constitution.md` and includes
   an updated Sync Impact Report comment at the top of the file.
2. The PR description MUST state the proposed version bump and its
   rationale (MAJOR / MINOR / PATCH per the rules below).
3. The PR MUST update or explicitly mark as unaffected every dependent
   template in `.specify/templates/` and any runtime guidance docs.
4. Merge requires approval from the project owner.

**Versioning policy** (semantic):

- **MAJOR**: A principle is removed, redefined in a backward-incompatible
  way, or a new NON-NEGOTIABLE rule is added that invalidates existing
  code.
- **MINOR**: A new principle or section is added, or a principle is
  materially expanded with new mandatory guidance.
- **PATCH**: Wording, clarification, typo, or non-semantic refinement.

**Compliance review**:

- Every PR review is a compliance review. Reviewers MUST cite the
  principle (by Roman numeral) when requesting changes for a constitutional
  reason.
- A quarterly constitution review SHOULD be performed by the owner to
  confirm principles still match the project's needs and to retire any
  rules that have become obsolete.
